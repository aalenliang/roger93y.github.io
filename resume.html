2017.06 - 2017.08

在学习哈佛大学CS50 计算机入门课程时，课程第七周推荐 Flask 网络开发微框架，故购买《Flask Web开发：基于Python的Web应用开发实战》进行学习。
项目要求从零开始完成一个社交化的博客，要求有发布、阅读、评论文章，用户之间建立社交关系，权限管理等功能。
为了达成本书的目的且不至于了解的太浅，我搜集和借助了 Flask、Jinja2、SQLAlchemy、Alembic 等官方文档，作者项目的 issues 和 pull/request 列表，以及问答社区的帮助。
本项目是典型的MVC项目，对M而言，书籍使用 SQLite3，我更换为 MySQL，并解决遇到诸多配置问题。对C而言，Flask 的视图、路由、请求、RESTful API方面较为简洁，且有数量较多的开源库可以使用，对V而言，Jinja2 是强大的扩展性良好的模板。
本书读到第11章的时候我已经可以跳出作者的框架，通过官方文档进行自我定制，故本项目完成时和书本身的项目产生了一定的差异，这些差异来自我个人对博客的想法。
这个项目启发我用 Mac 自带的 Apache 服务建立了自己的文档库，并认识到软件有良好文档对软件本身意义重大，开发过程中碰到问题，应先重视官方文档，后查Stackoverflow等技术社区。

2017.07 - 2017.08

Flask Blog 项目阶段性结束准备开启一个新项目，作为使用7年多并从微博获益的人，我想复制一个微博，这个项目是对 Flask 加深理解和熟练的过程。

项目初期，阅读了微博架构相关文章，了解到微博是个量变引起质变例子，有着极其复杂的架构和极高的性能要求，复制两个字是天方夜谭，所以诚实谦虚的说模拟。我思考了微博的核心是传递信息，传递信息的主要表现有“写、评、转、赞、信”，这五个部分非常重要，参考了上个项目的经验，并且使用 Bootstrap 很快的搭建起了一个雏形。

实现转发和评论时候我希望能得到微博原生体验，减少页面刷新，思考后认为由后端路由传JSON，前端AJAX写DOM，思路是前端用fetch API获取JSON，然后用React的组件响应事件渲染，但实际操作需要补充的知识很多。

本着项目先跑起来的原则，我用后端路由先实现了“转、评”的项目原型。并手写了大部分 CSS ，以求让界面尽可能的接近微博，非常低效的体会到了设计师对 CSS 熟悉的同时，还要对页面的层级非常熟悉才能高效简洁的写出界面。

这个在建的项目让我意识到了RESTful API的重要性，阅读文章了解前后端分离的概念，后端MVC中的V是前端MVC/MVVM 的重要部分，后端根据前端的事件提供主要为JSON格式的数据，前端用JSON与后端交流并驱动视图。也对后端路由的使用情景进行思考，并意识到 RESTful API 对效率和团队合作有很大益处。虽然 Jinja2 已经足够应付一些情景，当需精益求精，提供良好用户体验，通过 API 进行数据交流让前后端分离，在页面内进行大多数事件是良好体验的保证，也是未来的趋势，或许也是消灭 APP 的必经之路。

2017.08 - 2017.08

该项目是我用 GitHub Pages + Jekyll 搭建的个人博客，是自己的个人主页。

想法的产生源于满足自己写东西需求，在项目中我想把自己待人待物的观点融入进去。

项目基于 GitHub Pages ，它提供免费的空间和的 [username].github.io 域名，且支持Jekyll 这个 Ruby 写的博客系统。网上的资料不计其数，但我不需要过多参考他人。吸取了之前 Flask Blog 的经验「开发过程碰到问题，应先重视官方文档，后查技术社区。」，所以按官方文档很快得到了一个默认主题的博客。

这个项目是把自己想法融入到应用中的实践。我对一个空白的默认主题进行了符合自己想法的定制，在设计的过程中体会到了 SASS 用变量定义 CSS 的高效、统一和标准化。

设计水平不高，所以多做减法，删减了非必要的和对视觉可能产生干扰的元素，希望更加专注于内容，以简单、轻量、美观为主。

提高阅读体验要做加法，我为 Jekyll 默认主题的 Markdown 添加了表格支持，优化了 Markdown 对中文的排版。考虑到移动端的阅读体验，我通过@media ，进行了简单的响应式设计，让移动端也能得到良好的排版。为减少阅读割裂感和提高便利性，文章底部我添加了前后文章的标题以快速跳转，并为移动端适配，当小屏幕访问时候将隐藏标题，只显示上一篇下一篇按钮。
